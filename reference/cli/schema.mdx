---
title: "openspec schema"
description: "Create and manage custom workflow schemas."
---

The `schema` subcommands let you create, fork, validate, and inspect workflow schemas. Schemas define the artifact flow for a change (e.g., `proposal → specs → design → tasks`).

## Subcommands

| Subcommand | Description |
|------------|-------------|
| `schema init <name>` | Create a new project-local schema. |
| `schema fork <source> [name]` | Copy an existing schema for customization. |
| `schema validate [name]` | Validate a schema's structure and templates. |
| `schema which [name]` | Show where a schema resolves from. |

---

## schema init

Create a new project-local schema.

```bash
openspec schema init <name> [options]
```

### Arguments

<ParamField path="name" type="string" required>
  Schema name in kebab-case (e.g., `research-first`).
</ParamField>

### Options

| Option | Description |
|--------|-------------|
| `--description <text>` | Schema description. |
| `--artifacts <list>` | Comma-separated artifact IDs (default: `proposal,specs,design,tasks`). |
| `--default` | Set as the project default schema. |
| `--no-default` | Don't prompt to set as default. |
| `--force` | Overwrite an existing schema with the same name. |
| `--json` | Output as JSON. |

### What it creates

```
openspec/schemas/<name>/
├── schema.yaml           # Schema definition
└── templates/
    ├── proposal.md       # Template for each artifact
    ├── specs.md
    ├── design.md
    └── tasks.md
```

### Examples

```bash
# Interactive schema creation
openspec schema init research-first

# Non-interactive with specific artifacts
openspec schema init rapid \
  --description "Rapid iteration workflow" \
  --artifacts "proposal,tasks" \
  --default
```

## schema fork

Copy an existing schema to your project for customization.

```bash
openspec schema fork <source> [name] [options]
```

### Arguments

<ParamField path="source" type="string" required>
  Name of the schema to copy.
</ParamField>

<ParamField path="name" type="string">
  Name for the new schema. Defaults to `<source>-custom`.
</ParamField>

### Options

| Option | Description |
|--------|-------------|
| `--force` | Overwrite an existing destination schema. |
| `--json` | Output as JSON. |

### Example

```bash
# Fork the built-in spec-driven schema
openspec schema fork spec-driven my-workflow
```

## schema validate

Validate a schema's structure and templates.

```bash
openspec schema validate [name] [options]
```

### Arguments

<ParamField path="name" type="string">
  Schema to validate. Validates all schemas if omitted.
</ParamField>

### Options

| Option | Description |
|--------|-------------|
| `--verbose` | Show detailed validation steps. |
| `--json` | Output as JSON. |

### Examples

```bash
# Validate a specific schema
openspec schema validate my-workflow

# Validate all schemas
openspec schema validate
```

## schema which

Show where a schema resolves from. Useful for debugging schema precedence when schemas exist at multiple levels.

```bash
openspec schema which [name] [options]
```

### Arguments

<ParamField path="name" type="string">
  Schema name to look up.
</ParamField>

### Options

| Option | Description |
|--------|-------------|
| `--all` | List all schemas with their sources. |
| `--json` | Output as JSON. |

### Example

```bash
# Check where a schema comes from
openspec schema which spec-driven
```

### Output

```text
spec-driven resolves from: package
  Source: /usr/local/lib/node_modules/@fission-ai/openspec/schemas/spec-driven
```

### Schema precedence

When a schema name exists at multiple levels, OpenSpec resolves in this order:

1. **Project** — `openspec/schemas/<name>/`
2. **User** — `~/.local/share/openspec/schemas/<name>/`
3. **Package** — built-in schemas bundled with the CLI

<Tip>
  Use `openspec schema which --all` to see every available schema and where it comes from.
</Tip>
