---
title: "Parallel Changes"
description: "Work on multiple changes simultaneously, context-switch mid-stream, and archive in bulk."
---

OpenSpec supports working on multiple changes at the same time. Each change lives in its own folder under `openspec/changes/`, so you can switch between them freely without losing progress.

## How parallel changes work

```text
Change A: /opsx:new ──► /opsx:ff ──► /opsx:apply (in progress)
                                         │
                                    context switch
                                         │
Change B: /opsx:new ──► /opsx:ff ────► /opsx:apply
```

Progress is tracked in `tasks.md` checkboxes. When you switch back to a paused change, `/opsx:apply` resumes where it left off.

## Example: context-switching mid-stream

You're working on one change and get pulled into an urgent fix:

```text
You: I'm working on add-dark-mode but got pulled into a bug fix

You: /opsx:new fix-login-redirect

AI:  Created openspec/changes/fix-login-redirect/

You: /opsx:ff

AI:  ✓ Created planning artifacts

You: /opsx:apply

AI:  Working on fix-login-redirect...
     [Implements the fix]
     All tasks complete!

You: /opsx:archive

AI:  ✓ Archived fix-login-redirect

You: Let's get back to dark mode

You: /opsx:apply add-dark-mode

AI:  Resuming add-dark-mode...
     Picking up at task 2.3: Update Header...
```

## Specifying change names explicitly

When multiple changes are active, specify the change name to target a specific one:

```bash
/opsx:apply add-dark-mode
/opsx:continue add-dark-mode
/opsx:verify add-dark-mode
/opsx:archive add-dark-mode
```

If no name is provided, commands infer from context—but explicit names remove ambiguity.

## Bulk archive for multiple completed changes

When several changes are done, use `/opsx:bulk-archive` to archive them all at once instead of archiving one by one.

### How bulk archive handles spec conflicts

When multiple changes touch the same spec files, bulk archive detects the conflict, inspects the codebase to confirm both are implemented, and merges specs in chronological order (by change creation date).

### Real bulk-archive example

```text
You: /opsx:bulk-archive

AI:  Found 3 completed changes:
     - add-dark-mode (all tasks done)
     - fix-login-redirect (all tasks done)
     - update-footer (all tasks done)

     Checking for spec conflicts...
     ⚠ add-dark-mode and update-footer both touch specs/ui/

     Inspecting codebase to resolve...
     Both changes are implemented. Will apply in chronological order.

     Archive all 3 changes?

You: Yes

AI:  ✓ Archived add-dark-mode
     ✓ Archived fix-login-redirect
     ✓ Archived update-footer
     Specs merged in order: add-dark-mode → update-footer
```

<Note>
  `/opsx:bulk-archive` is part of the expanded workflow. Enable it with `openspec config profile` then `openspec update`.
</Note>

## When to use parallel vs sequential changes

| Situation | Approach |
|-----------|----------|
| Urgent interrupt mid-feature | Parallel: create new change, finish it, resume original |
| Two independent features with no overlapping specs | Parallel: work on both, bulk-archive when done |
| Feature B depends on Feature A's code | Sequential: archive A first, then start B |
| Two changes that touch the same spec files | Parallel is fine—bulk-archive resolves conflicts automatically |
| Single developer, single focus | Sequential: simpler mental overhead |

<Tip>
  If changes touch overlapping specs, prefer sequential work unless there's a specific reason to parallelize. Bulk archive handles the merging correctly, but reviewing the merged result is easier when you do them in order.
</Tip>

## Best practices

- **Name changes clearly** so `openspec list` stays readable when multiple changes are active
- **Specify change names** in commands when context is ambiguous: `/opsx:apply add-dark-mode`
- **Verify before archiving** with `/opsx:verify` to catch drift between artifacts and implementation
- **Avoid too many parallel changes**—three or four is manageable; more than that and tracking becomes overhead
